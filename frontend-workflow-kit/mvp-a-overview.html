2어떻게 작동하나 — 데이터 흐름

사람/LLM은 frontmatter + 본문 표만 쓴다. 나머지(카운트·상태·판정)는 스크립트가 결정적으로 만든다.

불변식: 판정 로직은 readiness 한 곳 · 파생값(tbd_count 등)은 frontmatter에 안 쓰고 본문에서 계산 · 생성물엔 GENERATED 마커 · 같은 입력→같은 출력(멱등, 변동은 generated_at 한 줄만).

3readiness 모드 사다리 (누적)

바닥부터 올라가며 연속으로 충족되는 가장 높은 칸이 그 화면의 모드다. 높은 칸이 낮은 칸의 전제를 건너뛰지 못한다.

4파생값: 본문에서 계산 (수동 카운트 금지)

frontmatter에 tbd_count 같은 값을 적지 않는다 — 반드시 드리프트하기 때문. workflow-state가 본문/파일시스템에서 만든다.

7핵심 계약 — fake hook & AsyncState

“fixture→hook 교체 단계”를 제거한다. 화면은 처음부터 hook 하나에만 의존하고, 내부 구현만 단계에 따라 바뀐다.

8깨면 안 되는 불변식

  • 판정은 한 곳. readiness 판정은 readiness.mjs에만. 훅·스킬은 출력을 소비만 한다.
  • 파생값은 frontmatter에 안 쓴다. tbd_count 등은 workflow-state가 본문에서 계산.
  • 생성물엔 GENERATED 마커. 마커 밖은 생성기가 건드리지 않는다.
  • 사실의 단일 출처. zod=코드, query key=factory, props=컴포넌트. 문서는 링크/의도만.
  • 화면은 AsyncState 계약만 의존. TanStack Query 객체를 화면에 노출 금지.
  • confirmed 승격은 사람만. 승인 메타(approved_by/at/decision_id) 없으면 validate 실패.
  • 생성기는 멱등. 같은 입력 → 같은 출력. 변동은 generated_at 한 줄만.
  • 스크립트는 --json 지원. 의존성 최소(Node 내장 + yaml). 직접 실행 시에만 main().
