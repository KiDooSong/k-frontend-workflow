## Boundary held (adapter = discovery, core = determinism)

- The adapter only *discovers* hardcoded operation candidates; sorting, hook naming, output-path naming, rendering, and determinism are owned entirely by `codegen-core`. The unsorted discovery input + sorted golden is the witness.
- Output path patterns `clientOut`/`hookOut` stay Tier-1-role-derived (`{roles.api_client}` / `{roles.hook}`), so hook output remains domain-scoped `src/features/{domain}/hooks/**` (OD-7). Only the naming suffixes and client subdir/suffix differ, proving the conventions surface is config.

## Intentionally Not Done

- No registration in `scripts/adapters/codegens/manifest.json` — the fixture is loaded by `{module}`/path so the built-in resolution path stays byte-identical.
- No change to the existing `openapi-client` goldens / rendered files (golden immutability).
- No `check-generated-files` focused target and no `catalog/artifact-manifest.yaml` entry for the fixture — it is a `codegen-core` unit input, not a guarded artifact (isomorphic to the router dogfood being only a `route-core.test.mjs` input).
- No CI workflow, hard gate, required check, or `--enforce` promotion. No new role, preset, or package-script alias.
- No timestamps, prettier, or absolute machine paths (determinism preserved).
- OD-4 / OD-8 cross-check tooling and the CI promotion decision remain out of scope (separate follow-ups).
